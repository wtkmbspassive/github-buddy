Make the repositoryThe "repo" is just a folder that holds your project online
▸

A repository"Repository" (or "repo") is GitHub's word for a project folder. It holds all your files plus a history of every change. One project = one repo. is where your project will live. Let's make one.

  1. Go to github.com and log in.
  2. Top-right corner, click the + icon, then New repository.
  3. Repository name: type your-project-name. Names are specialGitHub names can't have spaces or most symbols. Spaces become hyphens, capitals usually become lowercase. This becomes part of your web address, so keep it short and clean. — see the box above for how yours looks.
  4. Description (optional): one sentence on what it does. Why botherThis shows up under your project name and in search results — the first thing visitors read. One clear sentence is plenty.. ✨ Need help? Open the description writer →
  5. Visibility: choose PublicPublic = anyone can see and download it. Private = only you and people you invite. For sharing a free tool, you want Public..
  6. Check Add a README file. What's thatA README is the "front page" of your project — the text people see when they open your repo. We write a proper one in Step 3. For now let GitHub create a blank one.
  7. For .gitignoreA .gitignore tells GitHub which files NOT to track. You only need it for complex coding projects. For a single file, leave it as 'No .gitignore'., leave it as No .gitignore.
  8. For LicenseA license says what people may legally do with your code. 'MIT' is the most common and permissive — basically 'do what you want, just don't sue me.' Great default for free projects., pick MIT License.
  9. Click the green Create repository.

Make a "Release"This creates the clean one-click Download button
▸

A ReleaseA 'Release' is a tidy official snapshot of your project, like 'version 1.0'. It gives visitors a clean download button instead of making them hunt through code files. gives people a friendly download button instead of forcing them to dig through your files.

  1. On your repo's main page, find Releases in the right sidebar → Create a new release.
  2. Click Choose a tagA 'tag' is a version label like v1.0. Type 'v1.0' and click 'Create new tag'. Future updates would be v1.1, v2.0, etc. → type v1.0 → confirm "Create new tag."
  3. Release title: something like v1.0 — First release.
  4. Describe this release: a few sentences on what it is and how to use it.
  5. Attach your file: drag your downloadable file into the "Attach binaries"'Binaries' just means 'files for download' here. Dragging your file here attaches it as a clean, directly-downloadable asset — the file people click to get. zone near the bottom.
  6. Leave the label as None (not pre-release).
  7. Click Publish release.
Now it's easy for everyone— just so you knowYour release page shows your file under "Assets" with a download link. In your README you can link to github.com/your-username/your-project-name/releases/latest so people always get the newest version.

Set up funding

Totally optional and not needed to publish. If you're sharing something useful for free, this lets grateful users chip in. Come back any time. The most GitHub-native option is GitHub Sponsors, but you can offer any combination.

Option 1 — GitHub Sponsors (most native, 0% fees)

GitHub's own funding tool. People sponsor you right from your profile or repo, with one-time or monthly amounts, and GitHub takes 0%GitHub charges no platform fee — they cover the processing costs. Behind the scenes it uses Stripe, so it's available in roughly 70 countries..

  1. Go to github.com/sponsors and click Join the waitlist / Set up Sponsors.
  2. Follow the steps to add your bank details (via Stripe) and set up optional tiersTiers are suggested amounts like $3/month or $25 one-time. You can also just allow custom amounts. Totally optional..
  3. Once approved, your profile gets a ♥ Sponsor button automatically.
Good to know— info, no action neededSponsors needs approval and isn't available everywhere yet. If you're not eligible or want extra options, the platforms below work for everyone and link from the same Sponsor button.

Whichever options you chose, this puts a ♥ Sponsor button at the top of your project linking them all.

  2. For the filename, type exactly: .github/FUNDING.yml (why the path?)Typing '.github/' first creates a special folder GitHub looks in. FUNDING.yml inside it is the exact name GitHub recognizes. Caps matter: FUNDING in caps, .yml lowercase.
  3. Paste this — keep only the lines for platforms you actually use, and replace yourname with your usernames:

Then commit it (message: Add funding links).

The part that catches everyone: creating FUNDING.yml is not enough by itself. On most accounts you also flip a setting, or the button silently never appears.

  1. In your repo, click Settings (top menu, gear icon).

  3. Find SponsorshipsThis checkbox controls whether the Sponsor button can display. FUNDING.yml says WHERE links go; this switch says SHOW them. You need both. and tick it. Saves automatically.
  4. Back on your repo's main page, a ♥ Sponsor button now appears.

Add a new filePut another file into your project alongside the others
▸

Adding more files later works exactly like your first upload — they sit alongside what's already there.

  1. On your repo's main page, click Add file → Upload files.
  2. Drag your new file(s) in (or click to browse). Give them a name that's different from your existing files, so they're added rather than replacing anything.
  3. Scroll down, write a commit message like Add project extra file.
  4. Keep Commit directly to the main branch selected, then click Commit changes.
Folders tooTo put a file inside a folder, type the folder name and a / before the filename when creating it (e.g. images/logo.png) — GitHub makes the folder for you automatically.
Want it in the download?If you made a Release (the clean download button), newly-added repo files aren't automatically in it. Edit the release (pencil icon) and attach the new file, or publish a fresh release.

Replace / update a fileUpload a new version over an old one — cleanly
▸

The trick: the filename must match exactly to replace rather than duplicate.

  1. Click Add file → Upload files (same as uploading).
  2. Upload your new file — with the exact same name as the one already there.
  3. Commit message: Update project to latest.
  4. Click Commit changes.
The #1 gotchaIf the filename differs even slightly (a dot vs underscore, a capital, "(1)" on the end), GitHub treats it as a new file and you get two. To check the exact name: click the file and read it in your browser's address bar — it never lies. Match that exactly.
Nothing is ever lostGitHub keeps every old version in history. You can't accidentally destroy your work.
Updating a release too?Replacing the repo file doesn't update a Release's attached download. Edit the existing release (pencil → remove old, attach new) or publish a new release with a higher tag like v1.1.

Delete a fileTidy up old or wrongly-named files
▸
  1. Click the file you want to remove, to open its page.
  2. Near the top-right, find the three-dots menu ⋯A '...' button hiding extra options. Some screens show a trash-can directly. If no trash can, click the three dots first. (or a trash-can 🗑️).

  4. Scroll down, keep the auto-filled message (or write Remove old file).

Safe to deleteBecause GitHub keeps full history, a deleted file can still be recovered from past commits. Deleting just removes it from the current version.
